Why this matters

1

The coverage gate is a number that doesn't enforce itself

Ginie defines a minimum of 5 submitMustFail cases. It generated 4. It flagged the shortfall. Then deployed. The gate exists as a check but carries no consequence — it's a linter with no CI attached.

2

Tests that don't compile provide zero coverage

The test scaffold failed to compile entirely. A compile failure means no test ran, no case was verified, no submitMustFail assertion was executed. The contract deployed with precisely zero test coverage despite the pipeline claiming otherwise.

3

Three gates, three bypasses

This is the third pipeline gate that logs a failure and deploys anyway — alongside the silent fallback and the audit HIGH findings. The pattern is consistent: sandbox policy treats every gate as advisory. Users are not told this upfront.

4

Proposed fix

Show test coverage as a metric on the results screen alongside Security / Compliance / Enterprise scores. A "Tests: 4/5 (scaffold failed to compile)" line on the dashboard would make the gap immediately visible without changing sandbox deploy behavior.
